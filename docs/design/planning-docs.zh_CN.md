# 中文策划资料导览与整合说明

本文用于补齐现有策划资料的中文版本入口。它不逐字替代每一份英文专项文档，而是把当前仓库里的设计文档、PRD、玩家入口说明和专项资料全部纳入中文策划索引，说明每份文档解决什么问题、当前应如何使用、和当前实现之间有什么关系。

最后核对日期：2026-06-20。

## 使用原则

- 当前实现事实优先看代码和 `docs/design/current-game-design.zh_CN.md`。
- 早期规格优先看设计意图，不直接当成当前实现清单。
- PRD 用于理解功能需求、验收标准和实施路径。
- 表现层文档用于图像、音频、图标、角色语气和资产方向，不直接改变模拟规则。
- 当文档与源码不一致时，重新核对源码，并在后续文档中标注偏差。

## 中文资料结构

新增中文资料分为两类：

| 中文文档 | 作用 |
|---|---|
| `docs/design/current-game-design.zh_CN.md` | 当前游戏完整设计总览。覆盖世界、剧情、等级、职业、战斗、任务、副本、经济、社交、PvP、表现和已知缺口。 |
| `docs/design/planning-docs.zh_CN.md` | 现有设计文档与 PRD 的中文导览。告诉读者每份原有资料的定位、重点和注意事项。 |

英文原文仍保留为细节来源，尤其是长表格、实现 hook、验收清单和专项方案。

## 当前版本总览

### `README.md`

定位：玩家和贡献者入口。

中文理解：

- 它说明 World of ClaudeCraft 是一个经典 MMO 风格微型游戏，也可作为无头强化学习环境。
- 它介绍在线托管、离线游玩、操作方式、当前副本、竞技场、社交系统和经典公式。
- 它是对外展示文档，适合快速了解游戏是什么、怎么玩、怎么跑起来。

策划使用方式：

- 用它检查玩家能否理解当前功能卖点。
- 不把它当完整策划表，因为它不会列出所有任务、怪物、数值和实现细节。
- 当中文策划总览和 README 有差异时，以代码和当前总览为准。

### `docs/design/current-game-design.md`

定位：英文当前设计总览。

中文对应：

- 中文版本为 `docs/design/current-game-design.zh_CN.md`。
- 两者描述同一件事：当前已经落地的游戏设计事实。

策划使用方式：

- 中文讨论优先读中文版本。
- 修改内容数据后，应同步更新中英文当前总览。

## 核心玩法与内容设计

### `docs/design/master-spec.md`

定位：6-20 级扩展的原始总规格。

核心设计点：

- 主线是三幕 Gravecaller 阴谋。
- 第一幕在 Eastbrook Vale，通过亡灵、Gravecaller 标记、Binding Rite 和 Hollow Crypt 引出 Morthen。
- 第二幕在 Mirefen Marsh，通过 Fenbridge、Drowned Chapel、Gravecaller 营地和 Sunken Bastion 对抗 Vael the Mistcaller。
- 第三幕在 Thornpeak Heights，通过 Highwatch、Wyrmcult、元素、复仇亡魂和 Gravewyrm Sanctum 对抗 Korzul the Gravewyrm。
- 文档包含 Zone 2、Zone 3、Hollow Crypt 增强、Sunken Bastion、Gravewyrm Sanctum、XP 预算、物品表和最低引擎变更清单。

当前关系：

- 这份文档的核心 6-20 主线已经成为当前内容骨架。
- 当前代码已经额外加入 Drowned Temple、Nythraxis 钩子、更多职业系统、Fiesta、市场和满级后成长。
- 因此它不是完整当前事实，只是理解 Gravecaller 主线和 6-20 级扩展设计意图的关键来源。

策划注意事项：

- 继续扩展主线时，应保持 Brother Aldric 和 Scout Maren 这种熟人回归结构。
- 副本被定义为剧情和装备事件，不是主要 XP 泡点。
- 新区域应继续遵循任务 XP、击杀 XP、重叠等级带和组队高点的平衡思路。

### `docs/design/spell-ranks.md`

定位：1-20 级技能等级表。

核心设计点：

- 九个职业都有经典风格技能等级进展。
- 技能等级使用真实经典学习等级和数值作为参考。
- 战士、法师、盗贼、圣骑士、猎人、牧师、萨满、术士、德鲁伊都有 1-20 的技能进阶说明。
- 文档强调尽量少加新引擎效果，优先复用现有效果类型。

当前关系：

- 它仍是技能等级和经典数值感的参考。
- 当前代码已经在它的基础上扩展了更多玩法，包括猎人宠物、术士多恶魔、德鲁伊形态、盗贼潜行毒药、天赋加成等。

策划注意事项：

- 新增技能或新等级时，先看这份文档，不要随意发明学习等级和数值。
- 需要区分“技能等级表参考”和“当前职业完整套件”。
- 当前完整套件请看 `current-game-design.zh_CN.md` 和 `src/sim/content/classes.ts`。

## 天赋与满级后成长 PRD

### `docs/prd/talents-and-specializations.md`

定位：天赋与专精系统 PRD。

核心设计点：

- 玩家从 10 级开始获得天赋点。
- 每个职业有天赋树，节点分为被动修正、主动技能授予和选择节点。
- 专精定义角色定位，例如坦克、治疗、输出，并提供标志能力和类似 Mastery 的被动。
- 支持免费洗点、多套 loadout、导入导出构筑字符串。
- 天赋效果要预计算成扁平修正，不能在战斗热路径每 tick 遍历树。
- 分阶段落地，先做一个职业切片，再扩展到九职业。

当前关系：

- 当前代码已经有九职业天赋树，每职业 26 个节点和三专精结构。
- 当前实现重点与 PRD 方向一致：服务器权威、loadout、构筑字符串、扁平修正和战斗锁定。

策划注意事项：

- 后续新增天赋效果时，必须先确认它能被现有修正结构承载。
- 避免无意义的强制填充节点，让构筑选择变成“数学唯一解”。
- 免费洗点和构筑分享是功能体验的一部分，不是附属项。

### `docs/prd/max-level-xp-overflow.md`

定位：满级后 XP 溢出与长期成长 PRD。

核心设计点：

- 20 级后 XP 不应被丢弃，而应进入 lifetime XP。
- lifetime XP 支撑虚拟等级、排行榜和外观里程碑。
- 满级后成长只提供炫耀与外观，不提供战斗力。
- 参考 RuneScape 的“等级上限不变，XP 继续累计”模型，以及经典 MMO 的 Paragon、Honor、Renown 等长期奖励节奏。
- 需要处理单人 XP、队伍 XP、任务 XP、UI、排行榜和持久化。

当前关系：

- 当前代码已经存在 lifetime XP、虚拟等级、威望消耗和里程碑数据。
- 当前中文总览记录了 250,000 到 5,000,000 lifetime XP 的里程碑。

策划注意事项：

- 满级后奖励必须保持横向，不得提供属性、伤害、掉落收益或经济优势。
- 排行榜和外观是该系统的主要保留目标。
- 低等级怪反复刷榜必须受 XP 灰名和等级差规则约束。

## 表现层与资产设计

### `docs/design/icon-system.md`

定位：程序化图标系统设计。

核心设计点：

- 技能、物品、buff、debuff 图标由程序化绘制生成。
- 目标风格是“经典 MMO 手绘图标”，包含背景、主体形状、调色板、发光、边框和品质色。
- 文档给出大量图标配方和图标 vocabulary。

当前关系：

- 文档中提出过多文件 `src/ui/icons/` 架构，但当前代码主要是扁平的 `src/ui/icons.ts`。
- 设计风格仍有效，但实现路径需要按当前代码核对。

策划注意事项：

- 新技能或新物品要考虑图标身份，而不是只给名字和数值。
- 图标应服务可读性：职业、法术学派、物品品质和效果类型要一眼能分辨。

### `docs/design/graphics-plan.md`

定位：画面升级实施计划。

核心设计点：

- 目标是从简单程序化画面升级到更有展示感的视觉层。
- 涵盖画质档位、后处理、IBL、地形材质、水体、天空、植被、角色和特效。
- 每一步都强调性能预算和低画质回退。

当前关系：

- 这是表现层规划，不改变模拟规则。
- 当前渲染仍要通过 `IWorld` 读状态，不能反向修改模拟。

策划注意事项：

- 画面升级不能破坏玩法可读性。
- 生物群系差异应服务区域身份：vale、marsh、peaks、crypt、sanctum、temple 都需要清晰视觉记忆点。

### `docs/design/lookdev-hookup.md`

定位：lookdev 接入渲染器的工程说明。

核心设计点：

- 说明天空、IBL、太阳锚点、后处理、地形、水体和 VFX sprite atlas 如何接入。
- 强调某些合同不变，例如太阳方向和渲染器接口。

当前关系：

- 它是图形工程集成笔记，不是玩法规格。

策划注意事项：

- 策划只需关心视觉结果是否支持区域氛围和战斗可读性。
- 实施时由渲染层按该文档和当前代码结构落地。

### `docs/design/ue5-overhaul-plan.md`

定位：资产与画面大升级计划。

核心设计点：

- 目标是用 CC0 资产、骨骼动画、PBR、IBL 等方式提高视觉质量。
- 文档列出资产来源、许可、视觉映射、管线和执行顺序。
- 明确模拟 key 不变，表现层按实体 kind、templateId 和 family 映射资产。

当前关系：

- `public/` 里部分资源来源与这份计划相关。
- 它仍是表现升级方案，不是玩法数值或内容设计。

策划注意事项：

- 替换资产时必须保持怪物家族和职业识别。
- 不能因为视觉升级改变碰撞、仇恨、技能命中或玩法逻辑。

### `docs/design/sound_effects.md`

定位：程序化音效目录与集成设计。

核心设计点：

- 音效覆盖移动、近战挥击、命中、防御、施法、投射物、治疗、光环、生物声和环境循环。
- 音效主要通过 WebAudio 程序化生成，而不是大量音频文件。
- 目标是强化反馈、空间感和经典 MMO 手感。

当前关系：

- 当前游戏已有程序化 WebAudio，用于战斗、UI、环境和奖励反馈。

策划注意事项：

- 新技能、新怪物机制和新副本首领应考虑音频提示。
- 音效应帮助玩家理解事件，而不是只增加装饰。

### `docs/design/npc_voices.md`

定位：NPC 声线和台词方向。

核心设计点：

- 按区域列出 Eastbrook、Mirefen、Thornpeak 和 Glimmermere Temple 的 NPC 声线。
- 包括 The Merchant、Marshal Redbrook、Brother Aldric、Warden Fenwick、Scout Maren、Captain Thessaly、Ondrel Vane 等。
- 每个 NPC 有语气、风格和可生成台词方向。

当前关系：

- 它是声音和写作参考，不表示运行时一定有语音播放。

策划注意事项：

- 写 NPC 文本时应保持角色一致性。
- Brother Aldric 和 Scout Maren 是跨区域叙事锚点，语气要稳定。

## AI 生命层专项设计

### `docs/design/ai-interactable-agents.zh_CN.md`

定位：NPC、怪物、物件和宠物接入 Codex worker 的主设计归档。

核心设计点：

- AI 生命层不是普通 NPC 聊天，而是给 NPC、怪物、物件、首领、宠物和场景加观察、记忆、语义和有限行动意图。
- 普通怪可以出现少量“奇点个体”，拥有临时个性、记忆、偏好和异常行为。
- 场景感知覆盖建筑、物件、天气、昼夜、掉落物、随行对象和区域危险压力。
- Codex 输出只作为表现和生命意图，任务、战斗、掉落、经济和副本权限仍由权威模拟和服务器命令决定。
- 文档已经记录已落地基线，包括 provider、场景语义、世界导演、审计、奇点记忆和主线风险规避。

策划注意事项：

- 新增 AI 能力时优先检查这份主方案，确认是否已有 profile、family、scene、item、world director 或 memory 路径。
- 任何新增玩家可见文案都必须走本地化路径，简中不能漏。
- 如果新增主动触发或轮询机制，请同时看下面的主动触发专题。

### `docs/design/ai-proactive-triggers.zh_CN.md`

定位：AI 生命层主动触发、事件队列和定时轮询机制的专题归档。

核心设计点：

- 补齐“玩家不点击也会发生”的世界呼吸：游戏事件触发和 5 分钟默认轮询并行。
- 设计一张轮询项目表，每个 rule 可配置周期、条件、候选对象、预算、provider 策略、冷却和审计标签。
- 覆盖饭点饥饿、附近物件想法、下雨躲避、晴夜星空、亡灵压力、守卫巡查、奇点个体主动生活、世界导演余波等场景。
- 强调只挑少数最相关对象主动反应，避免所有 NPC 或怪物同时冒泡。
- 给出 `AiActiveTriggerService`、主动事件队列、候选打分、预算、后台观测、测试和螺旋式实现计划。

策划注意事项：

- 这份文档是后续实现主动 AI 的入口，不替代 AI 主方案。
- 主动机制更容易影响主线体感，因此必须同步验证任务、战斗、拾取、商人、副本、宠物和经济状态不变。
- 默认周期 5 分钟只是初始值，实际每个 rule 都应能单独调参。

### `docs/design/ai-audit-center.zh_CN.md`

定位：AI 审计、token、后台可观测和运营清理能力的设计归档。

核心设计点：

- 记录 AI provider 调用、prompt 和 raw output 长度、延迟、P50/P90/P95、决策链路、memory writes、world director state 和 profile coverage。
- Admin Usage 页是当前 AI 生命层的主要观测入口。
- AI 主动触发机制上线后，也应把 rule、cursor、候选、跳过原因和安全拦截纳入该后台体系。

策划注意事项：

- 新增 AI 行为时不要只看游戏内是否有反馈，也要确认后台能解释“为什么发生”和“为什么没发生”。
- 清理 AI memory 时要理解它只清理 AI overlay 和审计记忆，不删除角色事实状态。

## Web3 与外观 PRD

### `docs/prd/woc/wallet-link.md`

定位：非托管钱包绑定 PRD。

当前状态：已实现于 v0.11。

核心设计点：

- 玩家可以通过签名消息把 Solana 钱包绑定到账户。
- 服务器保存账户到钱包的验证映射，只读读取链上状态。
- 游戏不托管密钥、资金或资产。
- 钱包绑定是可选的，不绑定也能完整游玩。

策划注意事项：

- 钱包能力是其他 $WOC 机制的基础设施，不是直接玩法优势。
- 所有 token utility 必须限制在外观、便利性、访问或服务器运营相关，不得提供战斗力。
- `src/sim` 必须保持纯净和确定性，不能引入链上状态作为模拟决策。

### `docs/prd/woc/holder-cosmetic-flair.md`

定位：持有者外观标识 PRD。

当前状态：已实现于 v0.11。

核心设计点：

- 验证过的钱包如果达到配置的 $WOC 持有阈值，可获得纯外观层级展示。
- 表现可能出现在姓名板、检查面板、玩家卡片等身份展示位置。
- 余额读取是只读的，外观不影响战斗、奖励、成长或经济。

策划注意事项：

- 外观身份能制造社交需求，但必须保持非战力。
- 公开展示只使用已验证绑定钱包余额，不使用未验证连接钱包预览。
- 后续要明确最终阈值、层级名称、刷新频率和资格是快照制还是连续持有制。
